## 3) Decisions Locked
- Product/technical decisions made this phase:
  - Phase 2 and Phase 3 should build on the existing benchmark `blame_map` contract, not introduce a second finding-focus mechanism.
  - Localized feature IDs and anchors are the right intermediate contract between feature recognition and finding focus.
  - A finding should only upgrade from raw violating-instance focus to localized-feature focus when the match is unambiguous for all violating instances in that finding; otherwise it should fall back to the old raw-instance path.
  - The first Phase 3 bridge is intentionally narrow: hole and pocket style findings only.
- Assumptions applied:
  - Existing frontend focus behavior in `DfmBenchmarkSidebar` is already capable of consuming smarter `blame_map.primary_anchor` payloads without a new UI contract.
  - Geometry inventory face bounds and localized feature anchors are reliable enough for face-overlap and bbox-containment matching in the first bridge slice.
- Defaults chosen:
  - Browser regression coverage continues to run from `web/tests/dfm-benchmark-overlay.spec.ts`.
  - Hole and pocket mappings are the initial high-confidence families for finding-to-feature blame maps.
  - When matching is ambiguous, the system preserves backward-compatible raw violating-instance blame maps.
